UI Widgets

The Expanded Widget

Think of Expanded as a greedy widget.
It takes up all the remaining empty space inside a Row or Column.

Row( children: [ Box, Expanded( child: Box ) ] )
Box width: 100
Expanded takes ALL remaining space
← fixed width → ← flex: 1 (fills everything) →
example.dart 
Row(children: [
  Container(width: 100, color: Colors.blue),
  Expanded(           // greedily fills the rest
    child: Container(color: Colors.green),
  ),
])
UI Widgets

ListView.builder

A regular ListView builds every item at once — it crashes with large lists. ListView.builder is smart: it only renders the items currently on screen.

Only visible items built ✓
home.dart 
ListView.builder(
  itemCount: items.length,
  itemBuilder: (context, index) {
    return ListTile(
      title: Text(items[index]),
    );
  },
)

State Management

The State Problem

If a button at the bottom needs data from the top, passing it through every widget in the middle is messy. We call this "Prop Drilling."

😩 Prop Drilling = Pain
🗄 Top Widget count = 5
Widget A count = 5
Widget B count = 5
👆 Button (needs count)

Every layer must accept and pass down the variable — even if it doesn't use it. This breaks as your app grows.

Provider Steps
1 Create The Brain

The ChangeNotifier

Create a class that holds your data. When something changes, call notifyListeners() — it shouts to every widget that's listening.

cart_provider.dart 
import 'package:flutter/material.dart';

class CartProvider extends ChangeNotifier {
  int itemCount = 0;   // our data lives here

  void addItem() {
    itemCount++;
    notifyListeners();  // 📣 shout to all listeners
  }
}
Provider Steps
2 Plug It In

The ChangeNotifierProvider

Wrap your app with ChangeNotifierProvider at the very top. Now every screen in the app can access the Brain.

main.dart 
void main() {
  runApp(
    ChangeNotifierProvider(     // 🔌 the plug
      create: (_) => CartProvider(),
      child: const MyApp(),
    ),
  );
}
Place it above MaterialApp so the whole widget tree has access.
Provider Steps
3 Listen To Data

context.watch — The Ears

Widgets that display data use context.watch. They rebuild automatically whenever the Brain calls notifyListeners().

cart_screen.dart 
class CartScreen extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    // 👂 watching — rebuilds when data changes
    final count = context
      .watch<CartProvider>()
      .itemCount;

    return Text('Items in cart: $count');
  }
}
Provider Steps
4 Trigger Actions

context.read — The Hands

Buttons that just change data — without needing to display it — use context.read. It's faster because it never rebuilds the widget.

product_card.dart 
FloatingActionButton(
  onPressed: () {
    // 👆 reading — calls method, NO rebuild
    context
      .read<CartProvider>()
      .addItem();
  },
  child: const Icon(Icons.add),
)
👂 context.watch — displays + listens
👆 context.read — acts, never rebuilds
